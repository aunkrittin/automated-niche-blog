# ถอดรหัส TCP Receive Window (RWIN) Auto-Tuning: ปรับแต่งระบบคำนวณบัฟเฟอร์เครือข่ายเพื่อลดค่า Latency และแก้ปัญหาปิงแกว่ง

เคยสงสัยไหมว่า ทำไมเน็ตบ้านความเร็วระดับ 1 Gbps ของคุณถึงยังมีอาการ "ปิงแกว่ง" (Ping Jitter) หรือมีค่า Latency พุ่งสูงขึ้นเป็นช่วงๆ ทั้งที่ไม่ได้ใช้งานอะไรรุนแรง? 

ในฐานะ System Architect และ Game Server Developer ผมมักจะพบว่าปัญหาคอขวดไม่ได้เกิดจากท่อส่งข้อมูล (Bandwidth) ที่แคบเกินไป แต่เกิดจาก **"การบริหารจัดการบัฟเฟอร์ (Buffer Management) ที่ไม่มีประสิทธิภาพ"** บนระบบปฏิบัติการของฝั่ง Client และ Server และตัวการสำคัญที่ทำงานอยู่เบื้องหลังเรื่องนี้ก็คือ **TCP Receive Window (RWIN) Auto-Tuning**

บทความนี้จะพาคุณไปเจาะลึกการทำงานของ RWIN แบบหมดเปลือก พร้อมเผยวิธีการปรับแต่งระบบเครือข่ายในระดับ Low-level เพื่อรีดประสิทธิภาพสูงสุด ลดค่า Latency และแก้ปัญหาปิงแกว่งให้หายขาด

---

## 1. RWIN คืออะไร? และมันส่งผลต่อ Latency อย่างไร?

ในการรับส่งข้อมูลผ่านโปรโตคอล TCP (Transmission Control Protocol) ซึ่งเป็นแบบ Connection-oriented และเน้นความน่าเชื่อถือของข้อมูล (Reliability) ตัวรับ (Receiver) และตัวส่ง (Sender) จะต้องมีการตกลงกันว่า "จะส่งข้อมูลได้มากแค่ไหน ก่อนที่ผู้รับจะต้องส่งสัญญาณตอบกลับ (ACK - Acknowledgment)"

**TCP Receive Window (RWIN)** คือ ขนาดของบัฟเฟอร์ (Buffer Size) ในฝั่งผู้รับที่เตรียมไว้สำหรับเก็บข้อมูลที่ยังไม่ได้ประมวลผล หากส่งข้อมูลมาเต็มความจุ RWIN นี้แล้ว ผู้ส่งจะต้อง "หยุดส่ง" เพื่อรอให้ผู้รับส่ง ACK กลับมาบอกว่าเคลียร์บัฟเฟอร์เรียบร้อยแล้ว จึงจะส่งข้อมูลชุดต่อไปได้

```
[ Sender ]  -----( Data: Window Size )-----> [ Receiver Buffer (RWIN) ]
    ^                                                    |
    |<---------------( ACK packet )----------------------|
```

### Bandwidth-Delay Product (BDP): หัวใจของการคำนวณ RWIN
สูตรทางคณิตศาสตร์ที่เป็นตัวกำหนดขนาด RWIN ที่เหมาะสมที่สุดคือ:

$$\text{BDP (Bits)} = \text{Bandwidth (bps)} \times \text{Round-Trip Time (RTT in seconds)}$$

*   หาก **RWIN < BDP**: ท่อส่งข้อมูลจะว่างเปล่า (Underutilization) เพราะผู้ส่งต้องหยุดรอ ACK บ่อยเกินไป ทำให้ทำความเร็วอินเทอร์เน็ตได้ไม่เต็มสปีด
*   หาก **RWIN > BDP**: จะเกิดพื้นที่บัฟเฟอร์ว่างเปล่ามากเกินไป ซึ่งนำไปสู่ปัญหาที่เรียกว่า **Bufferbloat**

### Bufferbloat: ศัตรูตัวฉกาจของเกมเมอร์และ Real-time Applications
เมื่อ RWIN มีขนาดใหญ่เกินไป ระบบปฏิบัติการจะพยายามสะสมแพ็กเก็ตไว้ในคิว (Queue) มากเกินความจำเป็น ส่งผลให้แพ็กเก็ตใหม่ๆ ที่เป็นข้อมูล Real-time (เช่น ตำแหน่งตัวละครในเกม หรือข้อมูล Voice Chat) ต้องไปต่อคิวรออยู่ข้างหลังแพ็กเก็ตขนาดใหญ่ ส่งผลให้เกิดอาการ **ปิงแกว่ง, ค่า Latency สูงขึ้นเฉียบพลัน (Ping Spikes) และการตอบสนองช้าลงอย่างเห็นได้ชัด**

## 2. ทำความเข้าใจระบบ TCP Auto-Tuning ใน Windows

ในอดีต (ยุค Windows XP) ค่า RWIN จะถูกกำหนดไว้เป็นค่าสเตติก (Static Value) เช่น 65,535 Bytes ซึ่งไม่เพียงพอต่อเน็ตความเร็วสูงในปัจจุบัน ตั้งแต่ Windows Vista เป็นต้นมา Microsoft จึงได้แนะนำฟีเจอร์ **Receive Window Auto-Tuning** ซึ่งจะคอยคำนวณและปรับเปลี่ยนขนาด RWIN โดยอัตโนมัติตามสภาพของเครือข่ายและค่า RTT ในขณะนั้น

อย่างไรก็ดี อัลกอริทึม Auto-Tuning ของ Windows มักจะถูกปรับแต่งมาเพื่อเน้น **"Throughput" (ดาวน์โหลดได้เร็วที่สุด) มากกว่า "Latency" (ตอบสนองเร็วที่สุด)** ส่งผลให้ในบางสภาวะการเล่นเกมหรือการรับส่งแพ็กเก็ตขนาดเล็กแบบ Real-time เกิดอาการหน่วงโดยไม่จำเป็น

> [!WARNING]
> **ข้อควรระวังก่อนเริ่มการปรับแต่งระบบเครือข่าย**
>
> การปรับแต่งค่า TCP/IP Stack และ Windows Registry มีผลโดยตรงต่อการทำงานของระบบเครือข่ายบนคอมพิวเตอร์ของคุณ การตั้งค่าที่ผิดพลาดอาจทำให้ความเร็วอินเทอร์เน็ตลดลง หรือเกิดปัญหาเชื่อมต่อไม่ได้ชั่วคราว
> 
> **สิ่งที่ต้องทำก่อนเริ่มลงมือ:**
> 1. แนะนำให้ทำการสร้าง **System Restore Point** ของ Windows ไว้ก่อน เพื่อให้สามารถย้อนกลับค่าเดิมได้ทันทีหากเกิดปัญหา
> 2. บันทึกค่าดั้งเดิม (Default) ของระบบเก็บไว้เสมอก่อนการแก้ไขใดๆ

## 3. ขั้นตอนปฏิบัติ: การปรับแต่ง RWIN Auto-Tuning เพื่อลด Latency

เราจะมาตรวจสอบและทดลองปรับแต่งค่า TCP Auto-Tuning บนระบบปฏิบัติการ Windows ผ่าน Command Prompt (CMD) ในโหมด Administrator

### ขั้นตอนที่ 1: ตรวจสอบสถานะ TCP ปัจจุบัน
เปิด Command Prompt ด้วยสิทธิ์ Administrator (Run as administrator) แล้วพิมพ์คำสั่ง:

```cmd
netsh int tcp show global
```

สังเกตที่บรรทัด **"Receive Window Auto-Tuning Level"** โดยปกติค่าเริ่มต้นของ Windows จะเป็น `normal`

---

### ขั้นตอนที่ 2: ปรับเปลี่ยนระดับ Auto-Tuning ให้เหมาะกับการใช้งาน

ระดับของ Auto-Tuning Level มีทั้งหมดดังนี้:
*   `disabled`: ปิดการทำงาน RWIN จะถูกจำกัดไว้ที่ค่าเริ่มต้น (64KB) **(ไม่แนะนำสำหรับเน็ตความเร็วสูง เพราะจะทำให้ความเร็วตกอย่างมาก)**
*   `highlyrestricted`: ปรับขนาด RWIN อย่างระมัดระวังสูงสุด ป้องกันไม่ให้บัฟเฟอร์ใหญ่เกินไป
*   `restricted`: ปรับขนาด RWIN แบบจำกัด ช่วยลดโอกาสเกิด Bufferbloat ได้ดี
*   `normal`: ค่าเริ่มต้นของระบบ (เน้น Throughput สูงสุด)
*   `experimental`: สำหรับการทดสอบ (อาจทำให้ระบบไม่เสถียร)

#### แนวทางที่แนะนำสำหรับเกมเมอร์และระบบที่ต้องการ Low Latency:
จากการทดสอบในระบบปิดและสภาพแวดล้อมจำลอง (Lab Environment) การปรับระดับเป็น **`restricted`** หรือ **`highlyrestricted`** จะช่วยจำกัดขนาดบัฟเฟอร์ไม่ให้ปวมเกินไปเมื่อมีการรับส่งข้อมูลพร้อมกัน ส่งผลให้ค่าปิงมีความเสถียรขึ้นอย่างเห็นได้ชัด

**คำสั่งสำหรับการปรับเป็น Restricted:**
```cmd
netsh int tcp set global autotuninglevel=restricted
```

**คำสั่งสำหรับการปรับเป็น Highly Restricted (สำหรับผู้ที่ต้องการรีด Latency ขั้นสุด):**
```cmd
netsh int tcp set global autotuninglevel=highlyrestricted
```

*หากต้องการย้อนกลับไปใช้ค่าเริ่มต้นของ Windows:*
```cmd
netsh int tcp set global autotuninglevel=normal
```

---

### ขั้นตอนที่ 3: ปิดการทำงานของ Nagle's Algorithm (ลด Delay ของแพ็กเก็ตขนาดเล็ก)
Nagle's Algorithm คือระบบที่พยายามรวมแพ็กเก็ตขนาดเล็กหลายๆ แพ็กเก็ตเข้าด้วยกันเป็นแพ็กเก็ตเดียวก่อนส่งออกไป เพื่อลด overhead ของ TCP Header แต่สำหรับเกมออนไลน์ นี่คือหายนะที่ทำให้เกิด Delay การยิงคำสั่งในเกม

เราสามารถเข้าไปปิดระบบนี้ได้ผ่าน Windows Registry:

1. กดปุ่ม `Win + R` พิมพ์ `regedit` แล้วกด Enter
2. เข้าไปยัง Path: 
   `HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters\Interfaces\`
3. คุณจะพบกับโฟลเดอร์ที่เป็น GUID (เช่น `{12345678-xxxx-xxxx-xxxx-xxxxxxxxxxxx}`) ให้คลิกเลือกโฟลเดอร์ย่อยที่ตรงกับเน็ตเวิร์กการ์ดที่คุณใช้งานอยู่ (สังเกตโฟลเดอร์ที่มีคีย์ `DhcpIPAddress` หรือ `IPAddress` ที่ตรงกับ IP ของเครื่องคุณ)
4. คลิกขวาที่พื้นที่ว่างทางด้านขวา เลือก **New > DWORD (32-bit) Value** สร้างคีย์ใหม่ 2 ตัวดังนี้:
   *   คีย์ที่ 1: ชื่อ `TcpAckFrequency` กำหนดค่า Value data เป็น `1` (ระบบจะส่ง ACK ทันทีที่มีแพ็กเก็ตมาถึง ไม่ต้องรอคิว)
   *   คีย์ที่ 2: ชื่อ `TCPNoDelay` กำหนดค่า Value data เป็น `1` (ปิดการสะสมแพ็กเก็ตของ Nagle's Algorithm)

```
Registry Editor
 └── Tcpip
      └── Parameters
           └── Interfaces
                └── {Your-Active-Network-Card-GUID}
                     ├── TcpAckFrequency = 1 (DWORD)
                     └── TCPNoDelay = 1 (DWORD)
```

5. ทำการ Restart คอมพิวเตอร์เพื่อให้การตั้งค่าทั้งหมดมีผล

## 4. ยกระดับฮาร์ดแวร์: ปัจจัยสำคัญที่ซอฟต์แวร์ชดเชยไม่ได้

การปรับแต่งค่าซอฟต์แวร์และ TCP/IP Stack ภายในเครื่องระบบปฏิบัติการสามารถช่วยลด Latency ภายในเครื่องได้ **แต่หากโครงสร้างพื้นฐานทางกายภาพ (Physical Infrastructure) ของระบบเครือข่ายคุณไม่มีประสิทธิภาพเพียงพอ ปัญหาก็คอขวดก็จะย้ายไปตกอยู่ที่ตัวอุปกรณ์ฮาร์ดแวร์แทน**

การแก้ปัญหา Bufferbloat และปิงแกว่งอย่างยั่งยืน จำเป็นต้องได้รับการสนับสนุนจากฮาร์ดแวร์เครือข่ายที่มีคุณภาพสูง:

1.  **สายแลนเกรดอุตสาหกรรม (High-Quality Ethernet Cables):** หลีกเลี่ยงการใช้สายแลนไม่ได้มาตรฐานที่มักจะแถมมากับเราเตอร์ทั่วไป เพราะสายที่มีสัญญาณรบกวน (Crosstalk) สูงจะทำให้เกิด Packet Loss ส่งผลให้ TCP ต้องลดขนาด Window ลงและเกิดการ Re-transmission (ส่งซ้ำ) ซึ่งเป็นสาเหตุหลักที่ทำให้ปิงพุ่งขึ้นกระทันหัน แนะนำให้ใช้สายเกรด Cat6a หรือ Cat8 ที่มีชีลด์ป้องกันสัญญาณรบกวน (S/FTP)
    *   [ดูรายละเอียดอุปกรณ์หรือสั่งซื้อสาย LAN Cat8 คุณภาพสูงได้ที่นี่](https://shopee.co.th/search?keyword=lan%20cat8%20cable)

2.  **เราเตอร์ที่รองรับ SQM (Smart Queue Management):** หากในบ้านมีการใช้งานอินเทอร์เน็ตพร้อมกันหลายคน (เช่น มีคนดูสตรีม 4K ขณะที่คุณกำลังเล่นเกม) อัลกอริทึมจัดคิวทั่วไปบนเราเตอร์ราคาถูกจะไม่สามารถรับมือได้ การเปลี่ยนไปใช้เราเตอร์ระดับ Gaming หรือเราเตอร์ที่รองรับระบบจัดคิวแบบ **Fq_CoDel** หรือ **CAKE (Common Applications Kept Enhanced)** จะช่วยจัดการระดับคิวบัฟเฟอร์ที่ต้นทาง ส่งผลให้ค่าปิงในเกมเสถียรคงที่ แม้จะมีคนดาวน์โหลดไฟล์ขนาดใหญ่อยู่ในวงแลนเดียวกันก็ตาม
    *   [ดูรายละเอียดอุปกรณ์หรือสั่งซื้อ Gaming Router ประสิทธิภาพสูงได้ที่นี่](https://shopee.co.th/search?keyword=gaming%20router%20sqm)

## บทสรุป

การทำความเข้าใจและปรับแต่ง **TCP Receive Window (RWIN) Auto-Tuning** ควบคู่ไปกับการตั้งค่า Registry ของระบบเครือข่าย คือกุญแจสำคัญที่ทำให้ระบบปฏิบัติการ Windows ของคุณตอบสนองต่อแพ็กเก็ตข้อมูลได้อย่างรวดเร็วและเป็นระเบียบมากยิ่งขึ้น การเปลี่ยนผ่านจากโหมดเน้นความเร็วสูงสุดเพียงอย่างเดียว (`normal`) มาสู่โหมดที่เน้นความสม่ำเสมอของเวลาตอบสนอง (`restricted`) ร่วมกับการใช้ฮาร์ดแวร์ที่มีคุณภาพ จะช่วยให้คุณสัมผัสได้ถึงความลื่นไหลและ Latency ที่นิ่งสงบ ไม่ว่าจะเป็นการใช้งานทั่วไปหรือการแข่งขันในแมตช์สำคัญของคุณ!
