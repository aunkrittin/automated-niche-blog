# ถอดรหัส TCP Congestion Control ใน Windows: ปรับแต่งระบบ CUBIC และ BBR เพื่อเพิ่มเสถียรภาพเครือข่ายและลด Latency ขั้นสุด

ในโลกของระบบเครือข่ายที่มีความเร็วสูงและการตอบสนองที่รวดเร็วระดับมิลลิวินาที (Low Latency) ไม่ว่าจะเป็นการสตรีมมิ่งข้อมูลขนาดใหญ่, การทำธุรกรรมการเงินที่มีความถี่สูง (HFT), หรือการเล่นเกมออนไลน์แบบ Real-time ปัญหาที่ผู้ดูแลระบบและวิศวกรซอฟต์แวร์มักพบเจอคือ **"ทำไมอินเทอร์เน็ตแบนด์วิดท์สูง แต่ยังมีอาการหน่วงหรือแพ็กเกจสูญหาย (Packet Loss)?"**

คำตอบส่วนใหญ่ไม่ได้อยู่ที่สายสัญญาณทางกายภาพ แต่อยู่ที่กลไกการควบคุมความหนาแน่นของเครือข่าย หรือ **TCP Congestion Control Algorithm (CCA)** บนระบบปฏิบัติการ Windows ในบทความนี้เราจะเจาะลึกสถาปัตยกรรมการทำงานของสองอัลกอริทึมหลักอย่าง **CUBIC** และ **BBR** พร้อมทั้งเผยวิธีปรับแต่งค่า Registry และ PowerShell เพื่อรีดประสิทธิภาพเครือข่ายของ Windows ให้ถึงขีดสุดอย่างปลอดภัย

---

## 1. เข้าใจกลไก TCP Congestion Control: CUBIC vs BBR

ก่อนที่เราจะทำการปรับแต่งระบบ เราจำเป็นต้องเข้าใจก่อนว่าอัลกอริทึมทั้งสองตัวนี้มีหลักการคิดและแก้ไขปัญหาเครือข่ายที่แตกต่างกันอย่างสิ้นเชิง

```
[Sender] ---> (Congestion Window / Buffer) ---> [Bottleneck Link] ---> [Receiver]
```

### CUBIC (Loss-based Congestion Control)
CUBIC เป็นอัลกอริทึมมาตรฐาน (Default) ของ Windows 10/11 และ Windows Server มาอย่างยาวนาน 
*   **หลักการทำงาน:** CUBIC จะคาดการณ์แบนด์วิดท์ที่ใช้งานได้โดยการส่งข้อมูลเพิ่มขึ้นเรื่อยๆ เป็นกราฟเส้นโค้งกำลังสาม (Cubic Function) จนกระทั่งเกิด **Packet Loss** (ข้อมูลสูญหาย) ระบบจึงจะรับรู้ว่าเครือข่ายหนาแน่น จากนั้นจะลดหน้าต่างการส่งข้อมูล (Congestion Window - cwnd) ลงอย่างรวดเร็ว แล้วค่อยๆ ไต่ระดับขึ้นไปใหม่
*   **ข้อจำกัด:** ในเครือข่ายยุคปัจจุบันที่มีบัฟเฟอร์ขนาดใหญ่ (Bufferbloat) CUBIC มักจะส่งข้อมูลจนล้นบัฟเฟอร์ของ Router/Switch ทำให้เกิด Latency สูงขึ้นโดยไม่จำเป็น ก่อนที่จะตรวจพบ Packet Loss

### BBR (Bottleneck Bandwidth and RTT)
BBR เป็นอัลกอริทึมที่พัฒนาขึ้นโดย Google เพื่อแก้ไขข้อจำกัดของ Loss-based CCA
*   **หลักการทำงาน:** BBR จะไม่รอให้เกิด Packet Loss แต่จะคำนวณหาจุดสมดุลที่ดีที่สุดโดยการวัด **Max Bandwidth** และ **Minimum RTT (Round Trip Time)** อย่างต่อเนื่อง BBR จะส่งข้อมูลในปริมาณที่เท่ากับขีดความสามารถที่แท้จริงของเส้นทางเดินข้อมูล (Bottleneck) เท่านั้น
*   **ข้อดี:** ลดปัญหา Bufferbloat ได้อย่างมีประสิทธิภาพ ทำให้อัตรา Latency คงที่และต่ำที่สุด แม้ในขณะที่เครือข่ายมีโหลดสูงหรือมี Packet Loss ในสายสัญญาณ

---

⚠️ **[ประกาศเตือนภัยเพื่อความปลอดภัย / WARNING ALERT]**
การปรับแต่งค่า TCP Stack และ Registry ของระบบปฏิบัติการเป็นการเข้าถึงแกนกลางของระบบเครือข่ายในระดับ Kernel หากกำหนดค่าผิดพลาดอาจส่งผลให้การเชื่อมต่อเครือข่ายล้มเหลวหรือไม่เสถียร 

**ข้อแนะนำสำคัญก่อนเริ่มลงมือ:**
1. โปรดสร้าง **System Restore Point** ก่อนทำการแก้ไขทุกครั้ง
2. เปิดโปรแกรม PowerShell ด้วยสิทธิ์ผู้ดูแลระบบ (Run as Administrator) เท่านั้น
3. สำรองข้อมูล Registry Key ที่จะทำการแก้ไขเก็บไว้ก่อนเสมอ

## 2. การตรวจสอบสถานะ TCP Congestion Control ปัจจุบันบน Windows

ขั้นตอนแรก ให้เปิด PowerShell (Admin) แล้วรันคำสั่งด้านล่างเพื่อตรวจสอบว่าปัจจุบัน Windows ของคุณใช้งานอัลกอริทึมใดอยู่:

```powershell
Get-NetTCPSetting | Select-Object SettingName, CongestionProvider, AutoTuningLevelLocal
```

*ผลลัพธ์ที่ได้มักจะเป็น:*
*   `CongestionProvider` สำหรับโปรไฟล์ `Internet` มักจะถูกกำหนดไว้เป็น **CUBIC** หรือ **NewReno**

## 3. วิธีเปิดใช้งาน BBR (หรือ CUBIC แบบปรับแต่งพิเศษ) บน Windows

ตั้งแต่ Windows 11 (เวอร์ชัน 22H2 เป็นต้นไป) และ Windows Server 2022 เป็นต้นมา Microsoft ได้ใส่การรองรับ **BBR (BBRv1)** เข้ามาใน TCP Stack อย่างเป็นทางการแล้ว สำหรับเครื่องที่ต้องการเน้นความเสถียรและลดค่า Latency ในการใช้งานอินเทอร์เน็ตทั่วไปและการเล่นเกม การเปลี่ยนไปใช้ BBR ถือเป็นทางเลือกที่ยอดเยี่ยม

### ขั้นตอนการเปลี่ยน Congestion Provider เป็น BBR ผ่าน PowerShell

พิมพ์คำสั่งต่อไปนี้ลงใน PowerShell (Admin) เพื่อบังคับใช้ BBR บนโปรไฟล์การเชื่อมต่ออินเทอร์เน็ต:

```powershell
# ตั้งค่าให้โปรไฟล์ InternetCustom และ Internet ใช้ BBR
Set-NetTCPSetting -SettingName Internet -CongestionProvider BBR
Set-NetTCPSetting -SettingName InternetCustom -CongestionProvider BBR
```

*หมายเหตุ: หากระบบของคุณเป็น Windows รุ่นเก่าที่ไม่รองรับ BBR คำสั่งด้านบนอาจจะแสดง Error แนะนำให้สลับกลับไปใช้ CUBIC ซึ่งเสถียรที่สุดสำหรับ Windows รุ่นเก่าด้วยคำสั่ง:*

```powershell
Set-NetTCPSetting -SettingName Internet -CongestionProvider CUBIC
```

## 4. การจูนแต่ง TCP Stack เพิ่มเติมเพื่อลด Latency ขั้นสุด

นอกจากการเลือกใช้อัลกอริทึมที่ถูกต้องแล้ว การปรับพารามิเตอร์อื่นๆ ของ TCP Stack ใน Windows Registry จะช่วยเพิ่มประสิทธิภาพการทำงานร่วมกันได้อย่างมีนัยสำคัญ

### การปิดการทำงานของ Nagle's Algorithm (ลดความหน่วงในเกมและการส่งข้อมูลขนาดเล็ก)

Nagle's Algorithm มีหน้าที่รวบรวมแพ็กเกจขนาดเล็กแล้วส่งออกไปพร้อมกันเพื่อประหยัดแบนด์วิดท์ แต่ข้อเสียคือมันจะหน่วงเวลาส่งข้อมูล (เพิ่ม Latency) ซึ่งส่งผลเสียโดยตรงต่อเกมออนไลน์และโปรแกรม Real-time API

#### ขั้นตอนการตั้งค่าผ่าน Registry:

1. กดปุ่ม `Win + R` พิมพ์ `regedit` แล้วกด Enter
2. เข้าไปยัง Path ดังต่อไปนี้:
   `HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters\Interfaces\`
3. ภายในโฟลเดอร์นี้ คุณจะพบโฟลเดอร์ย่อยที่เป็นรหัส GUID (เช่น `{Subkey-Name}`) ให้หาโฟลเดอร์ย่อยที่เป็นการ์ดแลนหรือการ์ด Wi-Fi หลักที่คุณใช้งานอยู่ (สังเกตได้จากคีย์ที่มี `IPAddress` หรือ `DhcpIPAddress` ตรงกับเลข IP ของเครื่องคุณ)
4. คลิกขวาที่พื้นที่ว่างในโฟลเดอร์ย่อยนั้น เลือก **New > DWORD (32-bit) Value** และสร้างคีย์ใหม่ 2 ตัวดังนี้:
   *   ชื่อ: `TcpAckFrequency` ให้ดับเบิลคลิกและใส่ค่า Value data เป็น `1` (ระบบจะตอบรับแพ็กเกจทันทีโดยไม่รอ)
   *   ชื่อ: `TCPNoDelay` ให้ดับเบิลคลิกและใส่ค่า Value data เป็น `1` (ปิดการทำงานของ Nagle's Algorithm)

```
[Registry Path]
HKLM\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters\Interfaces\{Your-Interface-GUID}\
  ├── TcpAckFrequency = 1 (DWORD)
  └── TCPNoDelay      = 1 (DWORD)
```

## 5. การทดสอบและการวัดผลประสิทธิภาพ (Validation)

หลังจากที่คุณปรับแต่งระบบและรีสตาร์ตเครื่องคอมพิวเตอร์เรียบร้อยแล้ว แนะนำให้ทำการทดสอบผลลัพธ์ผ่านสองเครื่องมือนี้เพื่อยืนยันประสิทธิภาพ:

1.  **การทดสอบ Bufferbloat:** แนะนำให้เข้าไปที่เว็บไซต์วัดระดับ Bufferbloat เช่น [Waveform Bufferbloat Test](https://www.waveform.com/tools/bufferbloat) เพื่อวัดค่า Latency (Ping) ในขณะที่ดาวน์โหลดและอัปโหลดเต็มสปีด หากระบบได้รับการปรับแต่งอย่างถูกต้อง ค่า Latency ตอนอัปโหลด/ดาวน์โหลดควรจะใกล้เคียงกับช่วงที่เครือข่ายว่าง (Idle)
2.  **การใช้ iPerf3:** รันคำสั่ง iPerf3 ไปยังเซิร์ฟเวอร์ปลายทางเพื่อตรวจสอบแบนด์วิดท์จริงและอัตราการตกหล่นของแพ็กเกจ (Retransmission)

## 6. อุปกรณ์ฮาร์ดแวร์แนะนำเพื่อเสริมประสิทธิภาพเครือข่าย

การปรับแต่งซอฟต์แวร์ภายใน Windows จะแสดงผลลัพธ์ได้ดีที่สุดก็ต่อเมื่ออุปกรณ์ฮาร์ดแวร์ในระบบของคุณมีเสถียรภาพและสามารถรองรับปริมาณทราฟฟิกสูงๆ ได้อย่างไม่มีขวดโชม (Bottleneck) นี่คืออุปกรณ์ระดับโปรที่เราแนะนำเพื่อการเชื่อมต่อที่สมบูรณ์แบบ:

*   **สาย LAN คุณภาพสูงเกรดอุตสาหกรรม (CAT 8 / CAT 6A):** การส่งสัญญาณที่ไร้สัญญาณรบกวนจะช่วยลดอัตรา Packet Loss ทางกายภาพ ซึ่งส่งผลดีต่อการทำงานของทั้ง CUBIC และ BBR 
    *   [ดูรายละเอียดอุปกรณ์หรือสั่งซื้อสาย LAN เกรด Premium ได้ที่นี่](https://shopee.co.th/search?keyword=cat8+ethernet+cable)
*   **การ์ดเครือข่าย PCIe ความเร็วสูง (Intel Network Interface Card - NIC):** ชิปเซ็ตประมวลผลเครือข่ายของ Intel มีฟีเจอร์ TCP Offloading และลดภาระงานของ CPU ได้อย่างดีเยี่ยม เหมาะสำหรับเครื่องเซิร์ฟเวอร์หรือคอมพิวเตอร์ประสิทธิภาพสูง
    *   [ดูรายละเอียดอุปกรณ์หรือสั่งซื้อการ์ดจอและอุปกรณ์เน็ตเวิร์ก PCIe ได้ที่นี่](https://shopee.co.th/search?keyword=intel+pcie+nic)
*   **Router ระดับ Gaming/Enterprise ที่รองรับ SQM (Smart Queue Management):** หากเราจับคู่ Windows BBR เข้ากับเราเตอร์ที่มีระบบจัดการคิวที่ดี ปัญหา Bufferbloat จะหมดไปอย่างถาวร
